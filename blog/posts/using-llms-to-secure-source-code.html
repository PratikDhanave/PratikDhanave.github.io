Finding Vulnerabilities Is No Longer the Hard Part

By Pratik Dhanave · What Anthropic's playbook on AI-powered code security tells us about where the real bottleneck has moved
AppSecAI AgentsDevSecOps VerificationSecurity

For years, the scarce resource in application security was discovery. Skilled researchers spent days or weeks hunting for a single exploitable bug. That equation has flipped.

Anthropic just published a practical guide on using Claude Opus to secure source code, drawn from their work with security teams scanning both enterprise and open-source software. One number from it stopped me: as of late May 2026, their open-source scanning effort had disclosed nearly 1,600 vulnerabilities — but to their knowledge, only 97 had been patched.

Read that again. AI can now find vulnerabilities faster than the world can fix them. Discovery is easy to parallelize. Verification, triage, and patching are not. That's the new bottleneck, and it has real implications for how security teams should budget their time and money.

The find-and-fix loop

The teams getting the best results converged on a six-step workflow:

  1. Threat model first. Most false positives don't come from the model misreading code — they come from the model misunderstanding your trust boundaries. Is that config file attacker-controlled or trusted? Is that service internet-facing or internal? One CISO in the report summed it up perfectly: the model understands the code, but not the organization. Teams that documented their threat models (even as a simple THREAT_MODEL.md in the repo) saw exploitability rates on findings jump dramatically.
  2. Sandbox everything. Agents need strong isolation — both to protect your systems and to prove exploitability by actually running proof-of-concept exploits. Teams that let agents compile, test, and detonate PoCs in a sandboxed replica of production saw non-exploitable findings drop sharply.
  3. Discovery: less prompting, more context. Here's the counterintuitive part — long, prescriptive prompts make frontier models worse at finding bugs. Checklists narrow their creativity. What works: give the model the goal, the threat model, and good tools (grep, scanners, the ability to build its own tooling), then let it decide how to hunt. Partition the codebase and fan out parallel agents so they don't all converge on the same shallow bugs.
  4. Verification as a separate, adversarial step. Discovery should optimize for recall; verification for precision. Crucially, the verifier must be a fresh agent with no access to the discovery agent's reasoning — otherwise it just agrees. Prompt it to disprove each finding. Anthropic reports that an adversarial verifier roughly halved false positives, and requiring a working PoC brought them near zero.
  5. Triage by root cause. With models finding a hundred candidates before lunch, untriaged dumps destroy trust. Engineers stop reading reports — including the critical ones. Deduplicate by root cause, then score severity against concrete evidence: reachability, attacker control, preconditions, authentication, blast radius. Make the model reason through each question before assigning severity, or it anchors on the bug class and inflates the score.
  6. Patch with discipline. Write a failing test first, fix the root cause (not just the call site), hunt for variants, and run a fresh adversarial agent against the patch. A common failure mode: AI-generated patches that are so restrictive they break legitimate functionality. Minimal diffs win. And a human still owns the final review.

Why this matters for enterprise leaders

Three takeaways I'd highlight:

  • Budget for the pipeline after the scan, not just the scan. Your first AI security scan will surface far more findings than you expect. If you haven't built verification and triage capacity, you'll create alert fatigue, not security.
  • Context is the competitive advantage — again. Just as data fragmentation is the bottleneck for scaling agentic AI in the enterprise, missing organizational context (threat models, trust boundaries, architecture docs) is the bottleneck for AI security. The pattern repeats: model capability is rarely the constraint; your institutional knowledge, made machine-readable, is.
  • The defender's window is open — but it's a race. Anthropic is candid that it's getting easier for models to find and exploit vulnerabilities. The same capability serves attackers. The hopeful framing: for the first time, defenders can plausibly find and fix bugs before adversaries do — if they build the operational muscle now.

Anthropic has open-sourced a reference harness with skills for each step of the loop, so teams can try this on their own codebases today.

The era of "we can't find the bugs" is ending. The era of "can we fix them fast enough" has begun.
