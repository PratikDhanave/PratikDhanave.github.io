From Notebook to Production: What I Learned About Scaling AI Agents with ADK and GKE Autopilot

By Pratik Dhanave
GKE AutopilotKubernetesADK Workload IdentityProduction AI

Like many people experimenting with agentic AI right now, I've found that building an agent on a laptop is the easy part. The harder, less glamorous question is what happens after the demo works: how do you run that agent reliably, securely, and at scale, without turning your team into full-time infrastructure operators?

A recent Google Cloud tutorial by Abdel Sghiouar, "Scaling AI Agents: A Step-by-Step Guide to Deploying ADK on GKE Autopilot" (June 2026), offers a grounded answer to that question. I'm sharing my takeaways here — not as an expert pronouncement, but as notes from someone trying to learn this space alongside everyone else. I'd genuinely welcome corrections and alternative perspectives from those who have run agents in production longer than I have.

The gap between prototyping and production is real

Google's Agent Development Kit (ADK) makes it remarkably simple to scaffold an agent. In the tutorial's example, a basic "weather agent" — a Gemini-powered agent with a single Python tool function — is up and running locally in minutes, complete with a web UI for chatting with it.

But the tutorial is honest about something the industry sometimes glosses over: a working local prototype is not a production system. Production demands horizontal scaling, secure credential management, managed networking, and operational resilience. The guide's core argument is that GKE Autopilot — Google's mode of Kubernetes where nodes are fully managed for you — strikes a sensible balance between the flexibility of container orchestration and the simplicity teams actually need.

That framing resonated with me. Most organizations don't fail at agentic AI because their models are weak; they fail because the path from prototype to dependable service is underestimated.

Example of how lightweight the prototyping really is: the agent in the tutorial is about ten lines of Python — a get_weather(city) function that returns a string, passed as a tool to an ADK Agent configured with a Gemini model. Run uv run adk web, open localhost:8000, and you're chatting with it. In a real deployment, that hardcoded function would be swapped for an actual weather API call — and that's precisely the pattern: tools are just Python functions, so anything you can code becomes an agent capability.

Three architectural decisions worth understanding

Reading through the walkthrough, three choices stood out as broadly instructive, even if you never touch this exact stack.

1. Identity over API keys. Rather than embedding API keys in containers, the tutorial uses Workload Identity: a Kubernetes service account is mapped to a Google Cloud IAM service account, so the agent's pods can call Vertex AI (where Gemini runs) with scoped, auditable permissions and no long-lived secrets. This is a small configuration step with outsized security implications. If there's one practice I'd encourage any team to adopt early, it's this one — hardcoded credentials are the kind of shortcut that quietly becomes technical debt, then risk.

Example: the entire identity setup is just three gcloud commands — create an IAM service account, grant it the roles/aiplatform.user role on the project, and allow the Kubernetes service account to impersonate it via roles/iam.workloadIdentityUser. A single annotation on the Kubernetes ServiceAccount (iam.gke.io/gcp-service-account) then wires it together. No secret files in the image, no key-rotation calendar.

2. Owning the container lifecycle. ADK ships with a convenient one-command deploy to GKE, but the author deliberately avoids it for production, noting that the defaults (no Workload Identity, web UI exposed on a load balancer) aren't production-appropriate. Instead, the tutorial builds a multi-stage Docker image — a slim Python runtime with dependencies managed by uv — pushes it to Artifact Registry, and defines the Kubernetes Deployment, Service, and resource limits explicitly. The lesson generalizes: convenience tooling is wonderful for iteration, but production deserves deliberate, reviewable configuration.

Example: the Deployment runs two replicas with explicit resource requests (500m CPU, 512Mi memory) and limits (1 CPU, 1Gi), which is exactly what Autopilot uses to size and bill the workload. The agent itself runs headless as an API server (adk api_server) on port 8080 — no web UI in production — and you interact with it over REST: create a session, then POST a message to the /run endpoint and get structured JSON back. That API-first posture is what makes the agent embeddable in real products.

3. Modern ingress with the Gateway API. To expose the agent publicly, the guide uses the Kubernetes Gateway API (the successor to Ingress) with a Google-managed TLS certificate, a reserved static IP, and an explicit health-check policy. Certificates are provisioned and renewed automatically. Again, the broader takeaway is about reducing the operational surface area humans have to babysit.

Example: one gcloud compute addresses create reserves the global IP, one gcloud compute ssl-certificates create requests a managed certificate for your domain, and a single YAML file declares the Gateway (HTTPS on 443), an HTTPRoute forwarding to the agent's Service, and a HealthCheckPolicy probing /health. Roughly 20 minutes later — mostly certificate provisioning time — the agent is live at your own domain over HTTPS.

What the end state looks like

By the end of the walkthrough, you have an ADK agent running as a replicated Deployment on GKE Autopilot, authenticating to Gemini on Vertex AI through Workload Identity, reachable over HTTPS through a managed load balancer, and able to scale horizontally as demand grows. The agent itself is deliberately trivial — it returns a hardcoded weather string — because the point of the exercise is the scaffolding around it, not the agent logic.

I appreciated that the tutorial closes with cleanup instructions for tearing everything down to avoid charges. It's a small touch, but it reflects a practitioner's mindset: production thinking includes cost discipline.

Where this pattern applies beyond a weather demo

The weather agent is a stand-in, but the architecture maps directly onto real enterprise scenarios. A few examples that came to mind as I read:

A customer-support agent whose tools query your order-management system and knowledge base — Workload Identity scopes exactly which internal APIs it may touch. An IT operations agent that triages incidents by calling monitoring and ticketing tools, scaled out automatically during an outage when query volume spikes. A sales-research agent that enriches CRM records overnight, where the headless REST API makes it trivial to trigger from existing workflow automation. Or an internal analytics agent that answers natural-language questions over BigQuery — same Deployment, same Gateway, different tools.

In every case, the agent logic is the small part. The identity, scaling, and ingress scaffolding from this tutorial is the part that stays constant.

My honest reflections

A few thoughts I'm still sitting with:

The "infrastructure tax" on agentic AI is shrinking, but it hasn't disappeared. Autopilot removes node management; it doesn't remove the need to understand identity, networking, and resource sizing. Teams should budget learning time accordingly.

The hardest parts of production agents probably start after this tutorial ends — observability, evaluation, guardrails, session and memory management, cost controls on model calls. The guide itself points toward next steps like richer tools and multi-cluster resilience. I'd love to hear how others are approaching that layer.

And finally, a note of humility: I'm summarizing someone else's excellent work here, filtered through my own learning. If you're considering this path, read the original tutorial — it's a clear, step-by-step read of about 11 minutes, with all the commands and YAML you need.
