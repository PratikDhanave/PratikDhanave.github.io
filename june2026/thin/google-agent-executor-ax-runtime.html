Agent Executor (AX): Google's Open-Source Distributed Runtime for Production AI Agents

By Pratik Dhanave
Agent ExecutorDistributed SystemsGo Durable ExecutionMCP / A2A

The AI industry has spent the past two years racing to build agents. Now comes the harder problem: running them reliably in production. Google's answer is Agent Executor (AX) — an open-source, distributed agent runtime unveiled at agentexecutor.io and developed in the open at github.com/google/ax. Released under the Apache 2.0 license and currently in early preview, AX is Google's bid to define a runtime standard for agent execution, resumption, and distributed deployment.

The Problem AX Is Trying to Solve

Most agent frameworks today are excellent at one thing: orchestrating a reasoning loop inside a single process. That works for demos. It breaks down in production, where agentic workloads increasingly look like long-running distributed jobs — tasks that span minutes, hours, or even days, pause for human approval, call dozens of tools across networks, and inevitably hit failures along the way.

When a monolithic agent process crashes mid-task, the work is usually lost. There's no durable record of what happened, no way to resume from where things stopped, and little visibility into what the agent actually did. As organizations move from prototypes to autonomous, long-running workers, these gaps become showstoppers.

AX positions itself as the missing foundational layer: a runtime that coordinates agentic loops, logs every execution event durably, and communicates with both local and remote actors — so developers can focus on building agents rather than the infrastructure that keeps them alive.

What AX Actually Is

At its core, AX is a distributed runtime with a single-writer controller architecture. A central AX controller coordinates every user and agent call, maintains consistent state, and writes everything to a durable event log. Around the controller, the components of an agentic system — agents, skills, tools, and execution environments — run as isolated actors that can be deployed locally or remotely and spawned dynamically.

The architecture looks roughly like this: a client connects to the controller through a resumable stream (via a router), and the controller orchestrates communication with remote agents, MCP tool servers, and sandboxed environments that hold skills and built-in tools.

This design delivers several properties that matter in production:

1. Durable, Resumable Execution

Every execution is recorded in an event log (SQLite in the default local setup), giving AX automatic recovery from failures and interruptions — even across complex distributed deployments. If a client disconnects, it can replay missed events from the last sequence number it saw and catch up without rewinding the conversation. On compatible platforms, AX also supports compute-layer actor resumption, meaning the underlying workload itself can be suspended and revived.

2. Trajectory Branching (Forking)

Because all state lives in an event log, AX can fork a conversation from any checkpoint into a new event log using ax fork. Developers can branch alternate execution paths from a saved point without losing prior context — useful for experimentation, debugging, and "what if the agent had done X instead" analysis.

3. Auditing and Policy Control

Since every call flows through a common controller, AX provides a full audit trail, observability hooks, and trajectory collection for every execution. The built-in ax trace command spins up a local web UI that visualizes an execution's trace directly from the event log. For enterprises worried about what autonomous agents are doing on their infrastructure, this centralized chokepoint makes policy enforcement and auditing tractable.

4. Secure Isolation

Components run in secure-by-design sandboxes so that one misbehaving actor — say, an agent generating and executing code — can't cause harmful side effects across the system. Even the built-in bash tool requires explicit user approval through a confirmation flow before any command runs.

5. Model and Harness Agnosticism

AX deliberately avoids locking users into Gemini or any specific agent harness. You bring your own LLM and planning logic; AX handles execution, durability, and coordination underneath. A Gemini agent ships built-in (supporting both AI Studio API keys and Vertex AI authentication), but the runtime treats it as just another registered agent.

6. Open Protocol Support

AX natively speaks MCP (Model Context Protocol) for tools and A2A (Agent2Agent) for agent interoperability, alongside its own native AgentService gRPC interface. Examples in the repository show how to plug in remote agents, Google ADK (Python) agents, A2A agents through a bridge, and even experimental Colab-based agents running Python notebooks remotely.

The Developer Experience

Getting started is a one-liner for anyone with Go installed:

go install github.com/google/ax/cmd/ax@latest

The CLI revolves around a handful of commands:

  • ax exec — start or resume an agentic execution, optionally targeting a named agent (--agent coding) or a remote controller (--server localhost:8494). Conversations are identified by UUID and can be continued at any time with new input or resumed after a failure.
  • ax serve — run the controller as a gRPC server, configured through a YAML file (ax.yaml) that defines the event log backend, the planner model, skill directories, and a registry of remote agents.
  • ax fork — branch a conversation's event log from a checkpoint into a new conversation.
  • ax trace — visualize an execution trace in a browser.

Bringing your own agent is similarly straightforward: implement the AgentService gRPC interface, run it on an address, register it in ax.yaml under remote_agents, and the controller will route work to it.

A Worked Example: A Research Agent That Survives a Crash

To make this concrete, consider a team running a long-running "deep research" agent that might work for hours and occasionally needs human approval. Here's what that looks like end-to-end on AX.

Step 1 — Configure the controller. Everything starts with an ax.yaml that wires up the server, the event log, the planner model, and any remote agents:

server:
  address: ":8494"

eventlog:
  sqlite:
    filename: "eventlog/log.sqlite"

planner:
  gemini:
    model: "gemini-3.5-flash"
    timeout: "60s"
    skills_dir: "./examples/skills"

registry:
  remote_agents:
    - id: "medical-deep-researcher"
      name: "Medical Deep Researcher"
      description: "Performs deep medical research using resources like PubMed and clinicaltrials.gov"
      address: "localhost:50051"

Step 2 — Start the runtime and kick off work.

ax serve                        # Terminal 1: controller on :8494

ax exec --server localhost:8494 \
  --agent medical-deep-researcher \
  --input "Summarize recent phase III trials for GLP-1 agonists"

The controller assigns the execution a conversation UUID (say, d85a4b4e-...) and begins appending every event — model calls, tool invocations, intermediate results — to the durable event log.

Step 3 — Something goes wrong. Two hours in, the agent pod gets evicted, or your laptop loses Wi-Fi mid-stream. With a monolithic agent, the work is gone. With AX, you simply resume:

# Resume the incomplete execution from its last logged state
ax exec --conversation d85a4b4e-c53b-4c84-b879-f10d905bce40 \
  --agent medical-deep-researcher --resume

If only the client disconnected (the agent kept running), you catch up instead of rewinding — replaying just the events you missed:

ax exec --conversation d85a4b4e-c53b-4c84-b879-f10d905bce40 \
  --last-seq 12 --resume

Step 4 — Branch an alternative path. Suppose at sequence 12 the agent chose to focus on cardiovascular outcomes, but you also want to explore safety signals — without destroying the original trajectory. Fork from the checkpoint:

ax fork --src-conversation d85a4b4e-c53b-4c84-b879-f10d905bce40 \
        --dest-conversation e5e26e38-53a2-4f22-b1cb-ae867357df83 \
        --src-seq 12

ax exec --conversation e5e26e38-53a2-4f22-b1cb-ae867357df83 \
  --input "Instead, focus on adverse-event and safety data"

You now have two parallel trajectories sharing the same first 12 events — A/B testing for agent reasoning.

Step 5 — Audit what happened. When the work is done (or when compliance asks), visualize the full trace in a browser:

ax trace --conversation d85a4b4e-c53b-4c84-b879-f10d905bce40

Every model call, tool call, and approval gate is reconstructed straight from the event log.

Bonus: Bringing Your Own Agent in Three Terminals

The repository's remote-agent example shows how little glue is needed. A custom agent is just a gRPC server implementing AX's AgentService:

# Terminal 1 — your agent (e.g., a toy "lowercase" agent on :50051)
go run examples/remote_agent/main.go

# Terminal 2 — the AX controller (agent registered in ax.yaml)
ax serve

# Terminal 3 — route work to it
ax exec --server localhost:8494 \
  --input "HELLO, CAN YOU LOWERCASE WHAT I JUST SAID?"

Swap the toy agent for an ADK agent, an A2A-speaking agent, or a Colab-hosted Python notebook, and the controller-side workflow — durability, resumption, forking, tracing — stays exactly the same. That uniformity is the point.

Kubernetes and Agent Substrate

While AX is compute-agnostic and runs anywhere from a laptop to a large cluster, Google is explicit that the best experience targets Kubernetes. AX is designed to run on Agent Substrate, a companion open-source project built with the Google Kubernetes Engine team that extends Kubernetes for high-density agentic workloads — building on GKE's Agent Sandbox work around isolation, suspension/resumption, and job scheduling. For production, deploying AX on Agent Substrate is the recommended path.

This pairing reveals Google's broader strategy: just as Kubernetes became the standard substrate for containerized applications, Google wants the AX-plus-Substrate stack to become the standard substrate for agentic applications.

What AX Is Not

The project is unusually clear about its boundaries. AX is not:

  • A managed service — it's self-hosted, designed for your own clusters.
  • An agent framework — it doesn't compete with ADK, LangGraph, or CrewAI; it sits beneath them as the serving and durability layer.
  • A specific harness — it isn't a coding agent like Antigravity; it wraps harnesses and stays agnostic of their implementation.
  • A model-specific controller — any LLM can sit at the planning layer.

Roadmap and Current Caveats

AX shipped its first tagged release (v0.1.0) in May 2026 and remains in active early development. Google warns that the core, resumption protocols, and runtime specifications will see major breaking changes before a stable release, and external pull requests are paused during an ongoing architectural refactor — though bug reports and feature requests are welcome at ax-dev@google.com.

The published roadmap includes: Antigravity as the built-in harness, Bring-Your-Own-Harness (BYOH) support, suspension/resumption of subagents, tool-call approvals in subagents, and improved resumption protocols.

Why It Matters

AX arrives at an inflection point. Enterprises have proven agents can work; the unresolved question is whether they can run unattended, at scale, for days, safely, with an audit trail. Analysts note that a runtime alone doesn't solve governance questions like accountability and explainability of agent decisions — but it does strengthen the operational backbone those answers will require.

By open-sourcing the layer it built internally (with acknowledged influence from Google DeepMind's work on distributed harnesses), Google is betting that durable execution, isolation, resumption, and auditing become table stakes for every serious agentic application — and that AX becomes the neutral, protocol-friendly foundation the ecosystem standardizes on.

For developers, the takeaway is simple: if your agents are graduating from chat demos to long-running autonomous work, the infrastructure problems AX targets are ones you will hit. It's worth installing the CLI, running ax exec, and watching the event log do its thing.
