Google Just Put Graph Algorithms Inside the Database — Why Spanner Graph's New Capability Matters

By Pratik Dhanave
SpannerGraphGQL PageRankData Engineering

For years, enterprises have faced an awkward trade-off when working with connected data: either build complex ETL pipelines to ship data out to dedicated graph analytics engines, or run heavy analytical workloads on the operational database and watch transaction performance suffer. With the preview launch of graph algorithms in Spanner Graph, announced at Google Cloud Next, Google is betting that this trade-off no longer needs to exist.

What's New

Spanner Graph already unified relational and graph models in Google's globally distributed database, letting developers query connected data with the ISO Graph Query Language (GQL). The new release goes a step further: it brings Google Research's graph mining capabilities — the same lineage of work that produced PageRank — natively into the database.

The launch includes three families of algorithms:

  • Centrality — betweenness, closeness, and PageRank — for identifying the most influential nodes in a network.
  • Community detection — label propagation, modularity clustering, correlation clustering, weakly connected components, and clique aggregation — for surfacing hidden groups of tightly connected entities.
  • Similarity and path finding — Jaccard, cosine, common neighbors, and set-to-set shortest paths — for measuring relatedness and finding optimal routes.

The Architecture Is the Story

The algorithms themselves are classics. What's notable is how Google has packaged them:

  • No ETL, no second system. Algorithms are invoked directly in GQL and can be chained with standard queries — the output of an algorithm run can be written back into the graph and immediately queried. This collapses what used to be a multi-system workflow into a single query surface.
  • Isolation from production traffic. Execution happens on dedicated, automatically provisioned compute via Spanner's Data Boost, so analytical jobs don't compete with live transactions. Pricing is usage-based, which Google positions against the licensing costs of legacy graph analytics platforms.
  • Scale. Google claims the engine can process graphs with tens of billions of edges in minutes, thanks to a dense topology encoding optimized for random access.

A Concrete Example: Hunting Fraud Rings

Google's walkthrough of an anti-money-laundering scenario shows how the pieces fit together. Fraudsters typically operate networks of "mule" accounts that transact with each other. The workflow looks like this:

  1. Run modularity clustering across all accounts and write each account's community ID back to the graph.
  2. Query the graph to find which community has the highest concentration of already-flagged fraudulent accounts.
  3. Filter down to that suspicious community and run PageRank on just that subgraph to identify the most central account — the likely ringleader.
  4. Query the ringleader's recent outbound transfers to trace where the money went.

Each step alternates between an algorithm and a plain GQL query, all inside the same database. That interleaving — algorithm, query, algorithm, query — is the developer experience Google is selling.

Early Adopters

The customer list is telling. DaVita is using it for a Patient 360 view of healthcare networks. Yahoo is applying community detection and PageRank to audience segmentation across billions of unified user profiles. SoundCloud, which previously ran multi-hour batch graph jobs on custom clusters, is moving to the managed service to analyze its multi-billion-edge music graph on fresh data. WPP is using it to power graph analytics in its agentic marketing platform.

The use cases Google highlights beyond fraud — entity resolution, digital twins for network operations, recommendation systems, supply chain resilience, cybersecurity blast-radius analysis, and churn prediction — all share a common thread: the insight lives in the structure of the relationships, not in any single record.

My Take

This launch fits a broader pattern in enterprise data platforms: capabilities that once required specialized infrastructure are being absorbed into the operational database. We saw it with vector search; now it's happening with graph analytics. For data teams, the implication is fewer pipelines to maintain, fewer copies of data to govern, and analytics that run on current data rather than yesterday's export.

There are caveats. The feature is in preview and gated to Spanner's Enterprise and Enterprise+ editions, so it's aimed squarely at large-scale workloads. And "minutes on billions of edges" claims always deserve validation against your own data shapes. But for organizations already on Spanner — or those weighing a dedicated graph database against extending their existing stack — this meaningfully changes the calculus.

The interesting question for the next year: as graph algorithms, vector search, and full-text search converge inside one database, how much of the specialized analytics stack survives?
